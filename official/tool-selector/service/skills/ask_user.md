# 询问用户 领域知识

## 可用工具
| 工具 | 功能 | 参数 |
|------|------|------|
| `ask_user` | 向用户提问并等待回复，用于需要用户确认或补充信息的场景 | `question`(必填，问题内容)；`options`(可选，供用户选择的选项数组) |

## 工具选择指南
- 仅在**必要时**提问：缺少无法自行获取的关键信息、或动作不可逆需用户拍板（删数据、改生产、选方向）→ `ask_user`。
- 能自己检索、查文件、按合理默认推进的，**不要问**——自行决断并说明你的选择即可（与项目自主性偏好一致）。
- 有有限几个明确选项时传 `options`，让用户点选而非打字，减少来回。
- 范围变更、破坏性操作、多个等价方案需用户选 → 适合用本工具；琐碎选择（命名、格式）自己定。

## 协作流程
- 问题要清晰具体，**说明你需要用户提供什么**、为什么需要，必要时给出你的倾向性建议。
- 把相关的多个疑问合并成一次提问，避免连环单问反复打断用户。
- 拿到回复后按用户选择精确执行；用户明确选了某项就照做，不要再二次确认或自行改动。

## 常见陷阱
- 过度提问是最大的反模式：频繁打断会拖垮体验，每次提问前先自问"这个我能不能自己查/自己定"。
- 别问无意义的确认题（"我可以开始了吗""这样行吗"）——直接做，做完汇报。
- 问题含糊等于没问：用户看不懂你要什么就只能反问，反而多一轮。
- `options` 要互斥且完整，否则用户无从选起，还得自己补充说明。
